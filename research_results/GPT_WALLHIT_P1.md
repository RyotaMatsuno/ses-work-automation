Q2: #13 の Gemini フォールバック原因推測  
――――――――――――――  
現状「毎回 ERROR でフォールバック→実質 GPT-only」とのことですが、主に以下が疑わしいです。

1. DAILY_CALL_LIMIT 設定値  
   - コード上で10/日になっているが、SPEC は 30/日。実際に10回超えると API 側から 429（レートリミット）を返却 → ERROR。

2. モデル名の指定ミスマッチ  
   - “Gemini” ではなく “gemini-xyz” のように正確なエンドポイントを指定すべき。未登録モデル扱いでエラー。

3. API キー権限  
   - Gemini がベータ権限限定の場合、キーに該当スコープがないと 401 や 403 エラー。

優先度順は「呼び出し回数制限→モデル名→キー権限」です。まずログを掘って HTTP ステータス（429 vs 404/401）を確認してください。

――――――――――――――  
Q3: #18 スケジューラ二重共存はどちらを残す？  
――――――――――――――  
推奨は「Python scheduler.py」を残し、Windows Task Scheduler を廃止。理由：

- コードベースでバージョン管理・デプロイ一元化できる  
- クロスプラットフォーム対応（Linux/AWS Lambda など）にも応用可能  
- ファイルロック or プロセスロックもコード内で制御しやすい

Q4: 実装順序（2週間／Week4–5 想定）  
――――――――――――――  
以下の順で、上流のインフラ系→中盤ロジック系→後半大型改修→総合テストの流れがスムーズです。

  Day 5–6

Week5:  
  Day 1–3

      − フェーズ別ルーティング  
      − 呼び出し制限値修正  
      − Notion API システムプロンプト改善  
      − Geminiフォールバック検証

以上の並行・順次配置で、2週間以内に安全にリリース可能と見込みます。
