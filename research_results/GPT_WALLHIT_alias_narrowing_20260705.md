# GPT-5.4壁打ち: エイリアス絞り込み基準 (2026-07-05)

- **結論**
  - **基準案は概ね妥当だが、R1・R2・R7がやや粗く、R4・R5の運用定義不足が主リスク**です。
  - **suffix除去は C（2トークン分解）推奨**。スキル保持と職務要素保持の両立ができ、A/Bの欠点を避けやすい。
  - **R7=3回は暫定値としては妥当だが、件数確保用であり品質保証には弱い**。高危険パターンには頻度無関係の追加遮断が必要です。

- **1. R1〜R8の漏れ・過剰・リスク**
  - **R1 助詞含有除外**
    - **過剰除外リスク**あり。正常な複合語・表記ゆれに助詞様文字が混じるケースを誤殺する可能性。
    - ただし今回の失敗原因を見る限り、**文断片除去としては効果大**。
    - 改善案:
      - 「助詞を含む」だけでなく、**自然文パターン**として判定した方がよい。
      - 例: `.*(の|を|が|に|へ|と|や|も|は).*(経験|対応|担当|案件|募集|可能).*` のような**文断片寄り条件**を優先。
  - **R2 長さ上限**
    - **単純な文字数上限だけでは不十分**。
    - 短くてもゴミは入る（例: `aim`→`AI`）。
    - 逆に長くても正当なスキルはある（例: クラウド製品正式名、英語複合語）。
    - 改善案:
      - 文字数単独ではなく、**トークン数・空白数・記号率・日本語率**も併用。
  - **R3 数字・単価・括弧・記号断片除外**
    - **かなり有効**。
    - ただし**過剰除外**に注意。数字を含む正当スキルがある。
      - 例: `C#`, `.NET 8`, `Vue3`, `React18`, `Java17`, `RHEL8`
    - 改善案:
      - `万/円/月/～/〜/【】` など**求人ノイズ記号は強除外**
      - 一方で `C#`, `.NET`, `Node.js`, `Next.js`, `Vue3` などは**許可リスト**化
  - **R4 汎用canonical行き禁止ブラックリスト**
    - **必要だが、定義が曖昧**。
    - 「長文キー」の条件がないと運用者ごとに揺れる。
    - 「設計」「開発経験」などへの寄せを止めるのは正しいが、**では代替として何をするか**が必要。
    - 改善案:
      - **“汎用canonicalにはalias追加しない”を原則化**
      - 例外は既存精査済みaliasのみ
      - 「長文」の閾値を明示（例: **6字超 or 2トークン超**）
  - **R5 親スキルへの子サービスalias禁止**
    - **必須**。今回の品質崩壊の中核対策。
    - リスクは、**子サービスcanonical未整備のままOOVが増える**こと。
    - 改善案:
      - `AWS Glue` のようなものは **alias登録ではなく canonical追加候補キュー**へ送る
      - parent_skills機構が未実装なら、**禁止だけ先行**でよい
  - **R6 業界・職種・資格語除外**
    - **妥当**。
    - 漏れやすいのは、**職種っぽい技術語**との境界。
      - 例: `SRE`, `PMO`, `DBA`, `QA`
    - 改善案:
      - 除外対象は固定語彙だけでなく、**職種辞書/資格辞書で別レイヤ管理**
  - **R7 頻度閾値3回以上**
    - **ノイズ削減には有効**だが、precision対策としては限定的。
    - 同じ誤抽出が3回出れば通るため、**繰り返しノイズに弱い**。
    - また、**有用だが新しい/ニッチなスキルを落とす**。
    - 改善案:
      - 頻度は採否の主軸でなく、**レビュー優先度付け**に使うのが安全
      - 高危険パターンは頻度に関係なく落とす
  - **R8 20件サンプル人手レビュー**
    - **少なすぎる**。20件ではブレが大きい。
    - `false match 3/20超` は許容誤差15%で、alias品質としてはやや甘い。
    - 改善案:
      - 少なくとも **30〜50件**
      - かつ**層化サンプリング**（短英字、和文、クラウド系、略語系など）
      - 可能なら **“危険カテゴリ別に各10件”**確認

- **追加で入れるべきルール（漏れ）**
  - **R9: 短英字略語の厳格制御**
    - 例: `ai`, `aim`, `it`, `go`, `pm`
    - **3〜4文字以下の英字のみキーは原則禁止**、ただし明示許可リストで例外管理
    - これがないと `aim→AI` 系が再発しやすい
  - **R10: canonical側の受け入れ制約**
    - alias側だけでなく、**どのcanonicalに吸わせてよいか**を制限すべき
    - 例: `AI`, `AWS`, `Java`, `設計` など**高頻出・意味広すぎcanonical**には新規aliasを原則追加禁止
  - **R11: 完全一致/語境界チェック**
    - 部分一致由来の誤爆対策
    - `aim` と `AI`、`glue` と `AWS` のような**substring連想**を禁止
  - **R12: 正規化後差分の小ささチェック**
    - pre_normalize後に**意味が縮みすぎる**ものは危険
    - 例: `75万】java基本設計〜` → `Java`
    - 元文字列と正規化後の**情報落ち率が大きい場合は除外**
  - **R13: “経験/案件/募集/対応/担当/可能” など求人文脈語の除外**
    - 助詞だけでなく、**文断片の語彙シグナル**を使うと精度が上がる

- **2. suffix除去はA/B/Cどれか。理由1行**
  - **C** — **「Java」を救いつつ「運用保守」も落とさないため、情報損失と誤吸収のバランスが最良。**

- **3. R7の閾値3は妥当か**
  - **暫定的には妥当**。
  - ただし評価は **“件数を絞る閾値”として妥当**であって、**“品質を担保する閾値”ではない**。
  - 実務上は:
    - **危険パターンには無効**（3回出ても落とす）
    - **安全パターンには2回でも人手候補**に回してよい
  - つまり、**R7単独ではなく危険度別の可変閾値**が望ましい。

- **4. この基準で登録した場合の残存precision リスク上位2つ**
  - **1位: 短英字・略語の誤対応**
    - 例: `aim→AI` 型
    - R1〜R8だけだと、**短い英字ノイズが高頻度なら通る**可能性がある
  - **2位: 正規化で削りすぎた文断片の誤canonical化**
    - 例: `○○基本設計〜` → `Java` / `設計`
    - pre_normalizeとsuffix除去の結果、**求人文の残骸が“それっぽい単語”に縮退**して通るリスク

- **補足: この案をより安全にする最小修正**
  - **R9追加**: 3〜4文字以下の英字キー原則禁止、例外許可リスト制
  - **R10追加**: `AI/AWS/Java/設計/開発/経験` など高危険canonicalへの新規alias禁止
  - **R8強化**: 20件→**40件以上の層化レビュー**
  - **suffixはC** にして、`運用保守` は aliasではなく**職務タグ/補助属性**で保持するのが理想です。
