以下、**Recall@10 >= 85% / Precision@10 >= 40%** を狙いつつ、特に **0マッチ率 50% → 20%以下** を最優先にした改善案を10個に整理します。  
前提として、現状は **前段で落としすぎて母集団が消えている** のが最大ボトルネックなので、まずは「落とさない設計」に寄せるのが最重要です。

---

## 優先順位の考え方
- **S**: 最優先。0マッチ率改善に直結、かつRecall上限を押し上げる
- **A**: 高優先。S施策と組み合わせると効果大
- **B**: 中優先。精度安定化や取りこぼし削減に効く
- **C**: 低優先。補助的、または後追いで良い

---

# 改善案 10個

## 1. 前段ハードフィルタを「一致必須」から「加点式ソフトフィルタ」に変更
**優先度:** **S**  
**工数:** 中（3〜7日）  
**期待効果:**  
- **0マッチ率を大幅改善する最重要施策**
- 駅・経験年数・稼働日が全部揃う人材が **208人中65人のみ** という時点で、現行条件ではRecall上限が著しく低い
- まず候補集合を広げることで、**Recall@10を一気に改善**できる可能性が高い
- 0マッチ率は **50% → 20〜30%程度** まで下がる見込みがある

**具体案:**  
- 例: 駅一致、経験年数一致、稼働日一致を必須条件から外し、以下のようにスコア化
  - 必須: 法的/契約的に絶対必要な条件のみ
  - 加点: 最寄駅近い、経験年数近い、稼働開始近い、単価近い
- 「閾値未満で全落ち」ではなく、**最低でもTop-N候補は返す** 仕組みにする
- 例:
  - 完全一致: +1.0
  - 近似一致: +0.5
  - 不一致: 0
  - 総合点上位50件を後段Judgeへ

**リスク:**  
- 候補が増えて **Precision低下** のリスク
- 後段Judgeの負荷増
- ただし現状はRecallが致命傷なので、まずはここを壊すべき

---

## 2. 「0件回避フェイルセーフ」を実装し、候補ゼロ時は自動的に条件緩和
**優先度:** **S**  
**工数:** 小〜中（2〜5日）  
**期待効果:**  
- **0マッチ率を直接下げる**
- 現状の「厳しい条件のままゼロ終了」を防げる
- 実装コストの割に非常に効果が高い
- **0マッチ率 50% → 20%台** の有力施策

**具体案:**  
段階的に条件を緩める:
1. 駅一致必須を解除
2. 経験年数を±1年 → ±3年に拡張
3. 稼働開始を厳密日付 → 月単位に緩和
4. スキル一致率閾値を下げる
5. 最終的に最低件数K件までは返す

**例:**  
- まず厳しめ条件で検索
- 0件なら relax_level=1
- まだ0件なら relax_level=2
- 最低10〜30件出るまで緩和

**リスク:**  
- 緩和しすぎるとノイズ流入
- ただし「0件より多少ノイズのある候補」の方が、Recall目標には圧倒的に有利

---

## 3. スキル/職種/工程の alias 正規化強化（準OOV対策）
**優先度:** **S**  
**工数:** 中〜大（5〜10日 + 継続運用）  
**期待効果:**  
- **準OOVによる threshold 未達** を解消
- 同義語・略語・表記ゆれで落ちている候補を回収
- 0マッチ率低下にも効くし、**Recall/Precision両方に効く**
- 特にITスキルでは効果が大きい

**具体案:**  
- aliases辞書を強化
  - 例: Java Script / JavaScript / JS
  - AWS / Amazon Web Services
  - React.js / React / ReactJS
  - 基本設計 / 外部設計
  - PMO / プロジェクト推進
- N-gram, 半角全角, 大文字小文字, 記号除去
- よくある略称・誤記・和英混在を吸収
- 直近の0マッチ案件・取りこぼし案件から辞書を逆引きで増やす

**リスク:**  
- 誤正規化で別スキルが同一視される
- 辞書メンテ運用が必要
- ただし費用対効果は高い

---

## 4. 駅/沿線/勤務地マッチを厳密一致から「距離・沿線近接スコア」に変更
**優先度:** **S**  
**工数:** 中（4〜8日）  
**期待効果:**  
- 駅条件は0マッチの主要因になりやすい
- 厳密な駅一致をやめるだけで候補数がかなり戻る
- 特に都市圏案件では **Recall改善効果が大きい**
- 0マッチ率に対する寄与が高い

**具体案:**  
- 同一駅のみOK → 以下に変更
  - 同一駅: 高加点
  - 同一路線: 中加点
  - 乗換1回以内: 中加点
  - 同一都道府県/主要エリア: 低加点
- リモート可/併用可を別軸で強く加点
- 「最寄駅」ではなく「就業可能エリア」として扱う

**リスク:**  
- 通勤条件が厳しい案件でノイズ増
- 地理データ整備が必要
- ただし駅厳密一致は現状厳しすぎる可能性が高い

---

## 5. 経験年数の扱いを離散一致から「近似評価」に変更
**優先度:** **A**  
**工数:** 小〜中（2〜4日）  
**期待効果:**  
- 「3年以上」に対して「2.5年」「2年強」が落ちるような状態を改善
- 実務では年数はノイズが大きく、厳密一致の価値が低い
- Recall向上に効く
- 0マッチ率にも中程度寄与

**具体案:**  
- 例:
  - req=3年, candidate=5年 → 高加点
  - req=3年, candidate=2〜3年 → 中加点
  - req=3年, candidate=1年未満 → 低評価
- 経験年数は「必須判定」ではなく「適合度」に回す
- スキル有無 > 工程経験 > 年数 の順で重み付け

**リスク:**  
- 若手候補が混ざりPrecision悪化の可能性
- ただし後段Judgeで十分制御可能

---

## 6. 稼働開始日・稼働率・勤務条件を月単位/幅評価に緩和
**優先度:** **A**  
**工数:** 小（1〜3日）  
**期待効果:**  
- 稼働日厳格判定は0マッチ増加の定番要因
- 実運用では「来月から可」「○月中旬」「応相談」が多いため、日付厳密一致は不適
- 0マッチ率改善に効く

**具体案:**  
- 即日 / 今月 / 来月 / 2ヶ月以内 のようなバケット化
- 稼働率も 100% / 80%以上 / 週3可 など幅で扱う
- 応相談フラグを加点対象にする
- 必須条件は「絶対不可」だけ

**リスク:**  
- 交渉前提の候補が増える
- 営業現場で「会ってみると不可」が多少増える可能性

---

## 7. Judge v5.1 の閾値・判定基準を見直し、review→NG化の影響を緩和
**優先度:** **A**  
**工数:** 中（3〜6日 + 評価）  
**期待効果:**  
- 「旧review→新NG」の影響で候補が落ちているなら、後段でもRecallを削っている
- 前段緩和後にJudgeが強すぎると意味がないため、セットで見直すべき
- Recall改善に大きく効く可能性

**具体案:**  
- NG判定を3値化
  - OK
  - REVIEW
  - NG
- Top10生成では REVIEW を残す
- Judgeスコア閾値を再較正
- v5.1と旧版の差分分析を行い、何がNG化されたかをカテゴリ別に確認
  - 年数不足
  - 駅不一致
  - 文章表現差
  - 技術用語未正規化

**リスク:**  
- REVIEW流入でPrecision悪化
- ただし現状はRecall未達が確定しているため、まず緩める価値が高い

---

## 8. 「必須条件」と「希望条件」の再定義をデータモデルに明示
**優先度:** **A**  
**工数:** 中（4〜7日）  
**期待効果:**  
- 本来は加点でよい条件が、現在「暗黙の必須」になっている可能性が高い
- マッチングロジック全体の一貫性が増す
- 0マッチの構造原因を断てる

**具体案:**  
案件項目を以下に分類:
- **必須**: 法的・契約的・絶対不可
- **強希望**: なるべく満たしたい
- **希望**: あれば加点
- **参考**: 文章上の補足

例:
- 日本語必須、出社頻度上限、単価上限などは必須候補
- 駅、年数、開始日、業界経験は強希望〜希望へ

**リスク:**  
- 入力データ側の運用見直しが必要
- Notionや案件登録フロー変更が絡む可能性

---

## 9. 候補生成を複線化する：厳密検索 + 緩和検索 + embedding検索の統合
**優先度:** **B**  
**工数:** 中〜大（7〜14日）  
**期待効果:**  
- 単一ロジック依存を減らし、取りこぼしを回収
- aliasで拾えない類似案件/類似人材を救える
- Recall@10の底上げに有効

**具体案:**  
候補集合を複数経路から集める:
1. ルールベース検索
2. 緩和ルール検索
3. ベクトル検索（職務経歴書・案件文の意味類似）
4. 人気スキル/職種ブースト検索

その後に統合し、重複除去して再ランク

**リスク:**  
- 実装複雑化
- ベクトル検索がノイズを拾う可能性
- Precision維持には再ランキング品質が必要

## 10. 評価基盤を強化し、「0マッチ原因」を案件単位で可視化する
**優先度:** **B**  
**工数:** 中（4〜8日）  
**期待効果:**  
- 改善施策の当たり外れを早く見極められる
- 0マッチ率改善の再現性が上がる
- Recall/Precision最適化のPDCAが回しやすい

**具体案:**  
案件ごとに以下をログ出力:
- どのフィルタで何件落ちたか
- 0件になった最終原因
- alias適用有無
- JudgeでのNG理由
- 緩和レベルごとの候補数
- Top10に入った候補の特徴

ダッシュボード例:
- 0マッチ原因内訳
  - 駅 35%
  - 年数 20%
  - 稼働日 18%
  - スキル閾値 15%
  - Judge NG 12%

**リスク:**  
- 直接的な精度改善ではない
- ただし高速改善には必須に近い

# 目標達成への見立て

## Recall@10 >= 85%
現状の最大問題は、**前段で候補が消えすぎていること**です。  
したがって、Recall改善はまず「より賢いモデル」よりも、**候補を残す設計変更**が効きます。

特に、
- ハードフィルタ廃止
- 0件時の自動緩和
- 駅/年数/稼働日の近似化
- alias強化

このセットで、**Recallは最も伸びやすい**です。

## Precision@10 >= 40%
前段を緩めるだけだとPrecisionは落ちやすいですが、
- Judge v5.1見直し
- REVIEWクラス運用
- 再ランキング
- 必須/希望の整理

で十分に戻せる可能性があります。  
つまり、**先にRecallを取り、後でPrecisionを整える順番** が合理的です。
