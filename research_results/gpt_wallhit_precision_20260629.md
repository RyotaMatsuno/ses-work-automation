## 事実

## 最重要ポイント
もしこの「0matches」が**検索結果候補0件**を意味するなら、

### つまり

- **50%が候補ゼロ**

なら、**現定義のままでは目標未達が確定**です。  
これは単なる精度問題ではなく、**検索/前処理/フィルタ設計の構造問題**です。

---

## 0マッチの主因候補

### A. スキル以外のハードフィルタが厳しすぎる
エンジニア208人の充足率:
- スキル 96.6% → 約 **201人**
- 最寄駅 66.8% → 約 **139人**
- 稼働可能日 82.7% → 約 **172人**
- 経験年数 58.7% → 約 **122人**

特に弱いのが
- **経験年数 58.7%**
- **最寄駅 66.8%**

この2つを厳しめに使うと候補が急減します。

### 独立と仮定した概算
価格を除いて、  
スキル・駅・稼働日・経験年数が全部揃う確率は

- 0.966 × 0.668 × 0.827 × 0.587 ≒ **0.313**

208人中:
- 208 × 0.313 ≒ **65人**

つまり、**全条件をまともに比較できる人材は期待値ベースで約65人しかいない**。  
208人いるように見えて、**実質比較可能母集団は3割程度**の可能性があります。

これなら、案件条件が少し細かいだけで**0マッチ50%**は十分起こります。

---

- threshold 0.5

**期待効果:**  
- 駅条件は0マッチの主要因になりやすい
- 厳密な駅一致をやめるだけで候補数がかなり戻る
- 特に都市圏案件では **Recall改善効果が大きい**
- 0マッチ率に対する寄与が高い

**具体案:**  
- 同一駅のみOK → 以下に変更
  - 同一駅: 高加点
  - 同一路線: 中加点
  - 乗換1回以内: 中加点
  - 同一都道府県/主要エリア: 低加点
- リモート可/併用可を別軸で強く加点
- 「最寄駅」ではなく「就業可能エリア」として扱う

**リスク:**  
- 通勤条件が厳しい案件でノイズ増
- 地理データ整備が必要
- ただし駅厳密一致は現状厳しすぎる可能性が高い

---

## 5. 経験年数の扱いを離散一致から「近似評価」に変更
**優先度:** **A**  
**工数:** 小〜中（2〜4日）  
**期待効果:**  
- 「3年以上」に対して「2.5年」「2年強」が落ちるような状態を改善
- 実務では年数はノイズが大きく、厳密一致の価値が低い
- Recall向上に効く
- 0マッチ率にも中程度寄与

**具体案:**  
- 例:
  - req=3年, candidate=5年 → 高加点
  - req=3年, candidate=2〜3年 → 中加点
  - req=3年, candidate=1年未満 → 低評価
- 経験年数は「必須判定」ではなく「適合度」に回す
- スキル有無 > 工程経験 > 年数 の順で重み付け

**リスク:**  
- 若手候補が混ざりPrecision悪化の可能性
- ただし後段Judgeで十分制御可能

## 6. 稼働開始日・稼働率・勤務条件を月単位/幅評価に緩和
**優先度:** **A**  
**工数:** 小（1〜3日）  
**期待効果:**

- 実運用では「来月から可」「○月中旬」「応相談」が多いため、日付厳密一致は不適
- 0マッチ率改善に効く

**具体案:**  
- 即日 / 今月 / 来月 / 2ヶ月以内 のようなバケット化
- 稼働率も 100% / 80%以上 / 週3可 など幅で扱う
- 応相談フラグを加点対象にする
- 必須条件は「絶対不可」だけ

**リスク:**  
- 交渉前提の候補が増える
- 営業現場で「会ってみると不可」が多少増える可能性

## 7. Judge v5.1 の閾値・判定基準を見直し、review→NG化の影響を緩和
**優先度:** **A**  
**工数:** 中（3〜6日 + 評価）  
**期待効果:**  
- 「旧review→新NG」の影響で候補が落ちているなら、後段でもRecallを削っている
- 前段緩和後にJudgeが強すぎると意味がないため、セットで見直すべき
- Recall改善に大きく効く可能性

**具体案:**  
- NG判定を3値化
  - OK
  - REVIEW
  - NG
- Top10生成では REVIEW を残す
- Judgeスコア閾値を再較正
- v5.1と旧版の差分分析を行い、何がNG化されたかをカテゴリ別に確認
  - 年数不足
  - 駅不一致
  - 文章表現差
  - 技術用語未正規化

**リスク:**  
- REVIEW流入でPrecision悪化
- ただし現状はRecall未達が確定しているため、まず緩める価値が高い

## 8. 「必須条件」と「希望条件」の再定義をデータモデルに明示
**優先度:** **A**  
**工数:** 中（4〜7日）  
**期待効果:**  
- 本来は加点でよい条件が、現在「暗黙の必須」になっている可能性が高い
- マッチングロジック全体の一貫性が増す
- 0マッチの構造原因を断てる

**具体案:**  
案件項目を以下に分類:
- **必須**: 法的・契約的・絶対不可
- **強希望**: なるべく満たしたい
- **希望**: あれば加点
- **参考**: 文章上の補足

例:
- 日本語必須、出社頻度上限、単価上限などは必須候補
- 駅、年数、開始日、業界経験は強希望〜希望へ

**リスク:**  
- 入力データ側の運用見直しが必要
- Notionや案件登録フロー変更が絡む可能性

# 総論

- 面談化率
- 面談後成約率

# 各案の検証

## 1. ハードフィルタ→加点式ソフトフィルタ

### 期待効果
- 取りこぼし減少
- 条件が少しズレる候補も拾える
- 「実は会えば通る」人材/案件を出しやすい

### devil’s advocate
一番ありがちな失敗は、**“なんとなく近い候補”が大量に混ざる**ことです。

SESでは条件の一部は「加点」でよくても、  
一部は依然として**絶対条件**です。

例：
- 日本語ネイティブ/ビジネス必須
- 常駐必須 vs フルリモ不可
- 出社可能エリア
- 単価上限
- 外国籍可否
- 商流制限
- 年齢制限
- 稼働開始時期

これらまでソフト化すると、営業から見て  
**「見なくていい候補が増えた」**になります。

### 副作用
- **Precision低下**
- false positive増
- 営業レビュー工数増
- 「合ってないのに上位に出る」不信感

### 相互依存
- **3.alias正規化**や**4.距離スコア**が弱い状態でこれをやるとノイズが激増
- **5.経験年数近似**や**6.稼働日緩和**と同時に入れると、緩和が多重に効いて候補が緩すぎる

### 実装方式
- **ルールベースで十分可能**
- ただし「絶対条件」と「加点条件」の再定義が必要
- LLM不要

### 結論
**やる価値は高いが、“全部ソフト化”は危険。**  
推奨は：

- **Hard constraints層**
- **Soft scoring層**
- **Rerank層**

の3層に分けること。

---

## 2. 0件回避フェイルセーフ

### 期待効果
- 検索0件のUX悪化を防ぐ
- 営業に最低限の代替候補を返せる

### devil’s advocate
0件回避は**心理的に気持ちいいだけで、実務上は害にもなりうる**です。

0件は時に**正しい答え**です。  
無理に返すと：

- 「この条件でも出るんだ」と営業が誤認
- 顧客に無理筋提案
- 通過率悪化
- システム信頼低下

特にSESでは、  
**「該当なし」= ちゃんと条件を理解している**  
という価値もあります。

### 副作用
- false positive増大
- “無理やり出した候補”が営業工数を食う
- KPIを見誤る（件数だけ増える）

### 相互依存
- **1.ソフトフィルタ**導入後は0件自体が減るので、効果が薄まる
- **7.動的閾値調整**と似た役割があり、両方入れると過緩和になりやすい

### 実装方式
- ルールベース可能
- ただし返し方が重要  
  例：
  - 「完全一致0件」
  - 「条件を緩和した参考候補3件」
  と明示分離すること

### 結論
**“そのまま導入”は危険。**  
やるなら**検索結果に明確なラベル付け**が必須。  
「0件を隠す」のではなく、「0件だが近似候補を参考表示」が正しい。

---

## 3. alias正規化強化

### 期待効果
- Java Script / JavaScript / JS
- React.js / React / ReactJS
- AWS EC2 / Amazon EC2
のような表記ブレ吸収

### devil’s advocate
これはほぼやるべきですが、危険は**誤統合**です。

例：
- Java と JavaScript
- Spring と Spring Boot
- C と C#
- Vue と Nuxt
- Oracle と PostgreSQL を同列扱い

alias正規化は一歩間違うと、**Recallを上げる代わりに意味を壊す**。

### 副作用
- 誤正規化によるfalse positive
- メンテコスト
- 辞書更新運用の手間

### 相互依存
- **1.ソフトフィルタ**の精度土台
- **7.閾値調整**より先にやるべき
- **9.LLMリランカー**の前処理にも効く

### 実装方式
- **基本はルールベースで十分**
- 辞書 + 同義語 + 上位下位関係
- LLMは辞書候補作成補助には使えるが、本番判定はルール推奨

### 結論
**最優先級。**  
208人プールでも十分費用対効果あり。  
ただし「完全同義」「近縁」「関連」の階層を分けること。

---

## 4. 駅→距離スコア

### 期待効果
- 最寄駅一致だけでなく通勤現実性を評価できる
- 隣駅・沿線・乗換考慮でRecall改善

### devil’s advocate
これは見た目ほど簡単ではないです。  
SESで本当に重要なのは「駅名一致」ではなく：

- 実移動時間
- 乗換回数
- 路線相性
- 出社頻度（週1と週5で意味が違う）
- 客先常駐の心理的受容ライン

です。距離だけだと誤る。

例：
- 直線距離は近いがアクセス悪い
- 1回乗換20分のほうが、直通35分より嫌がられることもある

### 副作用
- スコア計算・データ整備コスト
- 距離スコアを過信すると“通える扱い”の誤判定
- 地方やバス前提地域で精度悪化

### 相互依存
- **1.ソフトフィルタ**で勤務地を加点化するなら重要
- **2.フェイルセーフ**の代替候補品質を上げる

### 実装方式
- **ルールベースで十分**
- まずは簡易版でOK：
  - 同一駅
  - 同一路線近傍
  - 乗換1回以内
  - 推定通勤時間帯
- LLM不要

### 結論
**高優先でよい。**  
ただし「距離」より**所要時間・乗換ベース**に寄せた方が実務的。

---

## 5. 経験年数→近似評価

### 期待効果
- 3年必須に対して2.5年や、4年を拾える
- スキルの濃さを踏まえた柔軟性が出る

### devil’s advocate
SESの経験年数は、そもそも**データ品質が低い**ことが多いです。

- 自己申告
- 案件単位で積み上げ誤差あり
- 触っただけでも経験に入る
- ブランクや古い経験が混ざる

よって、近似評価をやると  
**“数字がそれっぽい人”を過大評価**する危険があります。

また、案件側の「3年以上」は本当に3年必要なのではなく、  
**“若すぎると嫌” “一人称で動ける人”の代理指標**である場合が多い。  
そこを単純緩和するとズレます。

### 副作用
- false positive増
- 年数の見かけ倒しを拾う
- 古い経験を過大評価

### 相互依存
- **3.alias正規化**が弱いと経験年数集計も歪む
- **1.ソフトフィルタ**と併用すると候補増えすぎる
- **8.単価推定**にも影響しうる

### 実装方式
- ルールベース可能
- 例：
  - 必須3年に対し2.5年以上は減点付き通過
  - 直近2年以内の経験を重くする
  - “主担当/基本設計以降/リーダー経験”で補正
- LLMなくても可

### 結論
**有効だが、年数そのものを信用しすぎない設計が必要。**

---

## 6. 稼働日→月単位緩和

### 期待効果
- 8/1開始必須に対し8月中可などで拾える
- 実際の営業運用に近い

### devil’s advocate
これは現場実態に合う一方で、  
**月単位緩和しすぎると失注候補が増える**可能性があります。

SESでは開始時期はかなり重要です。特に：
- 交代枠
- 増員枠の予算消化タイミング
- 即日/上旬/中旬/下旬の違い

「8月案件」に対して8月下旬参画可を上位に出すと、  
結局営業が落とすだけです。

### 副作用
- false positive増
- “間に合わない候補”の混入
- 営業確認工数増

### 相互依存
- **2.0件回避**と役割が似る
- **1.ソフトフィルタ**に入れるなら重みは低めにしないと危険

### 実装方式
- ルールベース可能
- 例：
  - 月一致 = 高得点
  - 月内前半一致 = 中
  - 月内後半 = 低
  - 2か月先 = 基本除外
- LLM不要

### 結論
**実務的にはやるべき。**  
ただし“月単位OK”ではなく、**月内の前後ズレを段階評価**が良い。

## 7. SKILL_MATCH_THRESHOLD動的調整

### 期待効果
- 条件が厳しい案件では閾値を下げる
- 候補が多すぎる案件では閾値を上げる

### devil’s advocate
これは理屈はよいのですが、  
**運用がブラックボックス化しやすい**です。

営業から見ると：
- なぜこの案件は緩いのか
- なぜこの要員には厳しいのか
が説明できないと不信感が出ます。

また、母数208程度だと、動的調整で得られる恩恵が限定的な可能性があります。  
大規模推薦では有効でも、小規模では**固定ルール + 説明可能性**のほうが勝つことが多い。

### 副作用
- 一貫性低下
- デバッグ困難
- KPI悪化時の原因追跡困難

### 相互依存
- **1.ソフトフィルタ**導入後は、閾値そのものの意味が変わる
- **2.0件回避**と機能重複
- **9.LLMリランカー**を入れるとさらに説明性悪化

### 実装方式
- ルールベース可能
- ただし「動的」の定義を単純にするべき  
  例：
  - 候補0件なら必須以外の閾値だけ1段緩和
  - 候補20件超なら上位スキル一致を強化
- LLM不要

### 結論
**後回し推奨。**  
先に他の正規化・スコアリングを整えるべき。

### 相互依存

- LLMは本質解ではない

### 結論
**優先度は低め。**

# 1. 今すぐ（1–2日）改善 — 最大3つ

## ① alias正規化強化
**目的:** 0マッチの最大要因を最短で削る。  
**優先度:** 最優先

### Cursor作業指示
- `skill_normalizer` 相当の前処理層に、**技術スキルの別名辞書**を追加
- まずは**高頻度20–50語**だけに絞って実装
- 正規化は以下の3段階で処理
  1. 全角半角統一、英字小文字化
  2. 記号・空白揺れ除去（例: `C# .NET` / `C#.NET` / `C#`）

### 初期対象例
- `js` → `javascript`
- `ts` → `typescript`
- `node` / `node.js` → `nodejs`
- `react.js` → `react`
- `vue.js` → `vue`
- `aws ec2` → `aws`
- `gcp` → `google cloud`
- `postgres` / `postgresql` → `postgresql`
- `mysql8` → `mysql`
- `k8s` → `kubernetes`

### 実装ルール
- **誤統合しやすい語は除外**
  - 例: `java` と `javascript` は絶対統合しない
  - `spring` と `spring boot` は同一化しない
- 辞書はコード直書きではなく `yaml/json` 管理
- 正規化ログを保存
  - `raw_skill`
  - `normalized_skill`
  - `rule_hit`

### 完了条件
- 直近案件・人材データに対して
  - 正規化前後の**ユニークスキル数**
  - **0件率**
  - **正規化ヒット率**
  を比較できること

## ② 駅一致のハードフィルタをやめ、距離/沿線の加点式へ変更
**目的:** 勤務地条件による不必要な落選を止める。

### Cursor作業指示
- 現在の `station_exact_match == false なら除外` を停止
- 代わりに location score を追加
- まずは簡易版でよい

### スコア設計（初期版）

- 同一路線: `+0.7`
- 隣接駅 or 乗換1回圏: `+0.5`

- 駅マスタが無ければ、まずは簡易辞書で開始
  - `station -> line`
  - `station -> prefecture`

- UI/APIレスポンスに `location_match_reason` を返す
  - 例: `同一路線`
  - 例: `同一都県`
  - 例: `勤務地情報不足`

- 0マッチ案件率が改善すること

## ③ ハードフィルタを3層化（Hard / Soft / Rerank）
**目的:** Recall上限を物理的に引き上げる。

### Cursor作業指示
- 現在の前段filterを分解

### 新ルール
#### Hard（絶対条件のみ）
- 契約不可条件
- 稼働不可時期の完全衝突
- 必須資格など本当に必須なもの
- 法的/商流的に不可なもの

#### Soft（落とさない、加点減点のみ）
- 駅
- 経験年数
- 周辺スキル
- 勤務形態の細かい差
- 業界経験

#### Rerank
- Hard通過者に対し総合スコア順で並べる

### 実装上の注意
- **Hard条件を増やさない**
- configで各条件のレイヤを切り替えられるようにする

### 完了条件
- 「Hardで落ちた人数」「Softで減点された人数」が分かる
- 0件率が大きく下がる

## ④ alias辞書の一括大量投入
**やらない理由:**
- 誤統合リスクが高い
- 影響範囲の切り分けが難しい
- まず高頻度・安全な語から段階導入すべき

---

## ⑤ Hard条件の追加
**やらない理由:**
- 現状の根本問題に逆行する
- さらに0件率を悪化させる
- Recall目標85%から遠ざかる

# 4. 目標達成見込み時期

## 結論
**最短で2週間前後、現実的には3週間程度**です。

### 内訳
- **1–2日:** alias正規化 + 駅ハード除外停止 + 3層化
  - ここで0件率を大きく下げる
- **次の1週間:** 経験年数近似 + 稼働日緩和 + ログ/KPI整備
  - Recallをさらに押し上げる
- **追加1週間:** 閾値調整・誤統合修正・駅辞書補強
  - 85%到達ラインを安定化

### 期待値
- まずはRecall上限のボトルネックである0件率を潰すことで、
  **「未達確定」状態から「到達可能」状態へ移せる**
- ただし、**85%即達成保証ではなく、1回は閾値調整フェーズが必要**です

---

# 5. 監視KPI + アラート閾値

## 最重要KPI

### ① 0件率
**定義:** 全案件のうち候補0件で返る割合

- 目標: **15%未満**
- 警戒: **20%超**
- アラート: **25%超**

> 現状の最大課題。最優先監視。

---

### ② Hard除外率
**定義:** 全候補のうちHard条件で落ちた割合

- 目標: **30%未満**
- 警戒: **40%超**
- アラート: **50%超**

> Hardが厳しすぎるとRecall上限がまた死ぬ。

---

### ③ Recall@10
**定義:** 正解候補が上位10件内に入る割合

- 目標: **85%以上**
- 警戒: **80%未満**
- アラート: **75%未満**

> 主要目標そのもの。

---

### ④ Precision@10
**定義:** 上位10件中の妥当候補比率

- 目標: **現状維持以上**
- 警戒: **前週比 -10%以上**
- アラート: **前週比 -20%以上**

> Recall改善でPrecisionを壊しすぎていないか確認。

---

### ⑤ alias正規化ヒット率
**定義:** 全スキル語のうちalias辞書に当たった割合

- 目標: **10–30%程度**（初期はこれで十分）
- 警戒: **0–3%**
- アラート: **急に40%超**

> 低すぎると効いていない。高すぎると誤統合を疑う。

---

### ⑥ 参考候補発火率
**定義:** fallback_references が返った案件割合

- 目標: **10%未満**
- 警戒: **15%超**
- アラート: **20%超**

> 参考候補は補助機能。常用状態は危険。

---

### ⑦ 条件別落選理由の構成比
**定義:** 落選理由トップ3の比率

監視対象例:
- 駅
- 経験年数
- 稼働時期
- 必須スキル不足

- アラート条件:
  - 単一理由が**50%超**を占有
  - リリース後に特定理由が**前週比2倍超**

> 次にどこを触るべきか即分かる。
