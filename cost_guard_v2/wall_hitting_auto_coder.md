# 壁打ち: 自動コーディング装置 再構築方針

日時: 2026-06-17T13:01:27.365610

## 問題提起
# 問題
TERRA(SES事業)の自動コーディング装置 task_auto_runner が止まった。

## 経緯
1. task_auto_runner は pending_tasks/ に置かれた指示書(Cursor作業指示)を5分おきにスキャン
2. 拾ったら Claude Code CLI を `claude --dangerously-skip-permissions -p <指示書> --model sonnet --output-format json` で起動
3. 実装が走り、完了したら done/ に移動

## 詰まった理由
- Claude Code 2.1.144(2026年最新)が対話モード前提に進化
- `--dangerously-skip-permissions` は非対話モードでは Edit/Write を完全には抑止できない
- 結果: Claude Code が「ファイル書き込みの承認をお願いします」と発話して停止、ファイル0作成で正常終了($0コスト, 9.3分)
- 過去ログでは Claude Code 自身が「permission mode を auto-approve に変更が必要(UI 設定)」と言及

## 試して効かなかった対策
1. 指示書文面強化(承認禁止・即書き込み・計画のみ終了禁止など)
2. `ses_work/.claude/settings.json` で permissions.allow に Edit/Write/MultiEdit/Bash(*) を明示
3. 最初の書き込み対象を1ファイルに絞る(GPT-5.4 推奨)
→ 全て同じパターンで停止

## 当面の手動運用
CEO がCursor IDEを開いて指示書を Composer に貼り付けて実装させる
→ これは動く(動作実績あり)

## 求める助言
「松野は LINE/Claude だけで業務完結」設計に戻すため、対話なしで動く自動コーディング装置を再構築したい。

### 検討中の選択肢
A) Anthropic SDK 直叩きで自動コーディング装置を自作
   - Python から Anthropic API を直接呼び、ファイル読み書きツールも自前実装
   - メリット: CLI に依存しない、完全制御
   - デメリット: 改修自体に2-3日かかる、tool use のループも自前で書く必要
B) Aider など別の OSS コーディングエージェントに切り替え
   - aider-chat (Python製) は非対話モードで動く実績あり
   - メリット: 既存OSSで時間短縮
   - デメリット: 学習コスト、TERRA環境への適合
C) Claude Code CLI を古いバージョン(2.0系)にダウングレード
   - メリット: 過去動いていた可能性
   - デメリット: セキュリティ更新止まる、いつまで効くか不明
D) Claude Code CLI を捨てて、Cursor IDE の Composer を CLI 経由で叩く方法を探る
   - Cursor は対話なし実行 API を公開していない(2026-06 時点)
E) その他

## 評価観点
- 構築期間(短いほど良い)
- 維持コスト(低いほど良い)
- 「松野が触らない」設計への適合度
- 失敗リスク・回復可能性
- 月間費用($140/月の上限内)

# 1. 推奨案

## 推奨順位
1. **A) Anthropic SDK 直叩き** … 本命
2. **B) Aider 等 OSS エージェント** … 最短復旧の暫定策
3. **E) 併用設計** … 実務上はこれが最も安全
4. **C) Claude Code CLI ダウングレード** … 緊急避難なら可、恒久策には不適
5. **D) Cursor Composer のCLI化探索** … 非推奨

---

## なぜ A を本命にするか
今回の障害の本質は、**「自動化対象が対話前提の製品仕様変更に巻き込まれた」**ことです。  
したがって、恒久策は **UI/CLI製品の操作自動化ではなく、APIを直接制御すること** です。

Aの利点は以下です。

- **完全非対話で動く**
  - 承認待ちUIに詰まらない
- **停止条件を自分で決められる**
  - 「ファイル1つも変えず終了」を異常扱いにできる
- **権限境界を明示できる**
  - repo配下だけ書込可、危険コマンド禁止など
- **ログと再実行性が高い**
  - どのプロンプト・どの差分で失敗したか追跡可能
- **将来のモデル差し替えが容易**
  - Claude固定にせず、他社モデルも差し替え可能

### A の弱点
- 初期実装 2–3日は妥当
- ツールループ、差分適用、検証、リトライを自前で作る必要あり

ただし、今回必要なのは「汎用エージェント」ではなく、**TERRA向けの限定用途ランナー**です。  
そのため、**最小構成なら十分短期間で作れます**。

---

## なぜ B を暫定策に勧めるか
**業務停止コストが高いなら、Bで当日〜1日復旧を狙う価値があります。**

Aider のようなOSSは、
- 非対話で動かしやすい
- git差分前提で扱いやすい
- CLIとして安定しやすい

ので、**「今日から自動運転を再開したい」**には向いています。

ただしBは結局、
- OSS側の仕様変更
- TERRAの指示書との相性
- 実装品質のぶれ

に依存します。  
なので、**Bは恒久策ではなく“つなぎ”**と見るのが妥当です。

---

## C が非推奨な理由
**ダウングレードは延命であって解決ではありません。**

- 再現性が低い
- いつ壊れるかわからない
- セキュリティ・仕様差分を抱える
- 将来の再障害時に「またCEO手貼り」に戻る

もしやるなら、
- **48時間限定の暫定運用**
- **成功したらすぐA/Bに着手**
が前提です。

---

## D が非推奨な理由
**Cursor Composer は現時点で自動化基盤として不適**です。

- 非公開の内部仕様に依存しがち
- GUI操作の自動化は壊れやすい
- API非公開なら保守不能
- 「松野が触らない」設計に逆行

Cursorは**人間オペレータ向けの非常に良いUI**であって、  
**無人ジョブランナーの基盤**としては弱いです。

# 2. A を採用する場合の最小構成アーキテクチャ

結論：**「小さな専用エージェント」を作る**のが最適です。  
汎用IDEの再現は不要です。

---

## 最小構成
以下の 6 コンポーネントで十分です。

### 1. Task Watcher
役割:
- `pending_tasks/` を定期スキャン
- 1タスク1ワークツリーで処理開始
- 状態を `queued/running/review/done/error` で管理

実装:
- Python常駐プロセス or cron/systemd timer

---

### 2. Workspace Manager
役割:
- 各タスク用に repo を隔離
- 競合防止
- 失敗時ロールバック容易化

推奨:
- `git worktree add ../runs/<task_id>` を使う
- 各タスクごとにブランチ分離

利点:
- 並列実行しやすい
- diff回収が簡単
- 本体repoを汚さない

---

### 3. LLM Orchestrator
役割:
- Anthropic Messages API を呼ぶ
- ツール要求→実行→結果返却のループ
- 最大ターン数・最大コスト・最大時間で打ち切り

推奨ループ:
1. task指示書を読む
2. system prompt で制約を与える
3. model が tool call
4. Python側で tool実行
5. 結果を再投入
6. 最終的に「変更完了」「検証結果」を返させる

---

### 4. Tool Layer
最小で必要なのはこれだけです。

#### 必須ツール
- `read_file(path)`
- `write_file(path, content)`
- `list_files(glob?)`
- `search_in_files(pattern)`
- `run_command(cmd)`  
  - ただし許可制
- `git_diff()`

#### 強く推奨する制約
- 書込は **repo配下のみ**
- 削除禁止 or 明示許可制
- `run_command` は allowlist
  - 例: `pytest`, `npm test`, `ruff`, `mypy`, `go test`
- ネットワーク禁止
- シークレットファイル読込禁止

#### 重要
`write_file` より **`apply_patch` ツール**を作ると安定します。  
大規模ファイル全書換えより、差分適用の方が壊れにくいです。

---

### 5. Validator
役割:
- 「実装したつもり」を防ぐ
- 成果物ゼロ終了を失敗扱いにする

最低限の判定:
- diff が1行以上あるか
- 指示書の対象ファイルに変更があるか
- テスト/lint が通るか
- 生成ファイルが空でないか

失敗条件例:
- ファイル変更ゼロ
- モデルが承認要求や保留文言を出した
- 最大ターン数到達
- 検証コマンド失敗

---

### 6. Artifact/Notification
役割:
- `done/` へ結果格納
- LINE 通知
- 失敗時に人間が最小確認で再投入できるようにする

保存物:
- 最終diff.patch
- 実行ログ
- モデル会話ログ
- テスト結果
- コスト集計
- 失敗理由

---

## 推奨フロー
```text
pending_tasks/
  └ task_20260617_001.md

runner
  -> git worktree作成
  -> task内容をsystem/user promptへ投入
  -> tool loop開始
     -> read/search/list
     -> apply_patch/write_file
     -> run_command(test/lint)
     -> git_diff確認
  -> validator判定
  -> 成功: done/
  -> 失敗: error/
  -> LINE通知
```

---

## 最小 system prompt の考え方
ポイントは **「承認を求めるな」ではなく、「承認という概念を持たないツール環境にする」** ことです。

例:
- あなたは repo 内でのみ編集可能
- 編集は tool 経由で即実行する
- 計画だけで終了してはならない
- 変更が無い場合は失敗として理由を返す
- 最後に `summary / changed_files / tests / remaining_risks` を返す

CLI製品と違い、API直叩きでは**承認UIそのものが存在しない**ので、この方が効きます。

---

## コストを $140/月に収める設計
十分可能です。

### 抑制策
- 小さいモデルを標準にする
  - 普段は Sonnet 系
- 1タスク上限
  - 最大トークン
  - 最大ターン数
  - 最大実行時間
- 巨大ファイル全文投入を避ける
  - search/list → 必要箇所だけread
- prompt caching を使う
- まず差分だけ取らせる
- テストは対象限定
  - 全テストではなく changed scope 中心

### 目安
SES向けの中小タスクなら、  
**1タスクあたり数十セント〜数ドル程度**に抑えやすいです。  
月40〜100タスク程度なら設計次第で十分現実的です。

# 3. 見落としている選択肢

## E-1) 「完全自動」ではなく「自動実装 + 自動PR作成」に寄せる
もし本番反映まで無人にしなくてよいなら、  
**task -> branch作成 -> 実装 -> test -> PR作成 -> LINE通知** が非常に強いです。

利点:
- 失敗時の回復が簡単
- 差分レビュー可能
- 無人性と安全性のバランスが良い

TERRAの現実運用では、これが一番事故率が低いです。

---

## E-2) 「パッチ生成専用」に絞る
LLMには
- 変更方針
- unified diff
- 変更理由

だけを作らせ、  
実際の適用は Python 側のパッチ適用器でやる方式です。

利点:
- 書込事故が減る
- ログ追跡しやすい
- 検証に向く

これはAの実装をさらに簡略化できます。

---

## E-3) OpenHands / Sweep / Continue系のサーバ運用
候補にはなりますが、今回は優先度低めです。

理由:
- 依存が増える
- セットアップが重い
- TERRA専用要件には過剰

**「まず止血」**の局面ではやや大きすぎます。

---

## E-4) モデル二段構成
- Planner: 軽量モデル
- Editor: Sonnet
- Validator: 軽量モデル or ローカル検証

これでコスト圧縮できます。  
ただし初期版ではやりすぎです。**まず単一モデルでよい**です。
