# 【Cursor作業指示】DAILY_CALL_LIMIT 保留キュー化

## 対象ディレクトリ
ses_work/common/ledger.py, ses_work/mail_pipeline/mail_pipeline.py, ses_work/matching_v3/matching_v3.py

## 背景
DAILY_CALL_LIMITに到達すると、現在は処理を「破棄」している（stopped_call_limitとしてevent_logに記録するのみ）。
6月だけで2,261件の分類・マッチング処理が破棄され、機会損失が発生している。

## 作業内容
DAILY_CALL_LIMIT超過時の動作を「破棄」から「保留キュー化」に変更する。

### 1. common/ledger.py の reserve() 修正
- reserve()が上限超過でNoneを返す際、保留キューにエントリを追加する
- 保留キューはstate.sqlite3内の新テーブル `pending_queue` に保存
- スキーマ: id(INTEGER PRIMARY KEY), phase(TEXT), block_type(TEXT), target_id(TEXT), script(TEXT), queued_at(TEXT), status(TEXT DEFAULT 'pending')

### 2. mail_pipeline/mail_pipeline.py の修正
- allowed()がstopped_call_limitを返した場合、メールをprocessed扱いにせず、pending_queueに登録
- 次回のpipeline実行時にpending_queueから先に処理（FIFO）

### 3. matching_v3/matching_v3.py の修正
- CostGuard.can_call()がFalseの場合、案件をスキップせず pending_queue に登録
- 次回実行時に pending_queue の案件から先に処理

### 4. pending_queue の管理
- 翌日のDAILY_CALL_LIMITリセット後に自動再処理
- 7日以上保留された案件は自動失効（status='expired'）
- pending_queue の件数をログ出力

- pending_queue の件数がログに出力される
- 既存テスト(190件)が全て通る
