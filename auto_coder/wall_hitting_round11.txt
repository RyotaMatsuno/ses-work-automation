## Q1. 「案件全件取り込み」の最善アプローチ
**D案。Notion全件登録はやめる。全メールはまず生保存し、案件候補だけを段階分類する。**

### 推奨アーキテクチャ
1. **全受信メールを原文ごと永続化**
   - S3 / SQLite / Postgres のどれかに必ず保存
   - ここが「取りこぼしゼロ」の母集団になる

2. **一次判定は超広めの案件候補抽出**
   - 現行の `project / engineer / skip / other` をやめる
   - 代わりに  
     - `project_like`（案件の可能性あり）
     - `non_project`（広告、メルマガ、人材紹介、ノイズ）
   - に二値化する
   - **Recall最優先**。Precisionは捨てていい

3. **案件候補は全件構造化**
   - 事務、ヘルプデスク、キッティング、情シス、PMO補助、監視運用も含める
   - 「ITフリーランス案件」だけでなく「業務委託/派遣/SES案件」全般を取る設計にする

4. **Notion登録は全件ではなく絞る**
   - Notionは
     - `確度高`
     - `要確認`
     - `ノイズ混入候補`
   - のみ登録
   - それ以外はDB保存だけ

### 各案の評価
- **A案は非推奨**
  - 6,000件/日を全部構造化・Notion登録は無駄が大きい
  - ノイズで埋まる
  - Notion運用が死ぬ

- **B案は部分的に正しい**
  - 閾値緩和は必要
  - ただし「Notionに全部入れる」までやると破綻する

- **C案は中途半端**
  - skipもNotionに入れるのはNotionをゴミ箱化する
  - 保管先はNotionではなくDBにすべき

**結論:**
- **取り込み全件 = 原文DB**
- **案件候補全件 = 構造化DB**
- **Notion = 人が見る必要のある案件だけ**
- これが最適

## Q2. 6,000件/日を全件取り込むとコストどうなる？
**分類だけなら安い。全件構造化は件数次第だがまだ十分安い。問題はコストより運用負荷。**

### ざっくり試算
前提:
- 分類: **100 tokens/件**
- 構造化: **500 tokens/件**
- 6,000件/日
- Batch + nano 前提

### 1. 全件分類
- 6,000 × 100 = **600K tokens/日**
- nano + Batchなら**数十セント未満/日レベル**
- 現在 $0.04/日なら、分類を全件化しても**$0.1〜0.3/日程度**に収まる可能性が高い

### 2. 案件候補を構造化
案件候補数を仮置き:
- 300件/日 → 150K tokens/日
- 500件/日 → 250K tokens/日
- 1,000件/日 → 500K tokens/日

これも**大きなコストにはなりにくい**。

### 結論
- **$2/day制限にはまず収まる**
- 真のボトルネックは
  - Notion API
  - 人間の閲覧負荷
  - 誤登録の後始末
- **コストを理由に案件候補抽出を絞りすぎる必要はない**

## Q3. Notion DB側の設計変更は必要？案件6,000件/日は多すぎないか？
**必要。6,000件/日をNotionに入れるのは非推奨。Notionは最終ビューに限定する。**

### 理由
- 6,000件/日 = **18万件/月**
- Notionは大量明細の基盤ではない
- API 3 req/sec 制限もある
- 重くなる
- 重複やノイズ管理が地獄になる

### 推奨設計
#### 1. データ基盤を2層化
- **Raw Inbox DB**
  - message_id
  - account
  - received_at
  - subject
  - sender
  - body_text
  - body_hash
  - thread_id
  - source_mailbox

- **Structured Opportunity DB**
  - raw_mail_id
  - is_project_candidate
  - extraction_status
  - category
  - job_type
  - skills
  - unit_price_min/max
  - remote_type
  - nationality_rule
  - headcount
  - location
  - contract_type
  - confidence

- **Notion**
  - 高確度案件
  - 要確認案件
  - 営業で使う案件
  - 月次集計結果

#### 2. Notionのステータス設計
- `High Confidence`
- `Needs Review`
- `Archived Noise`
- `Duplicate`
- `Closed`

#### 3. 重複管理は必須
同じ案件が複数社から来るので、以下で近似重複判定:
- 件名正規化
- 単価帯
- スキル集合
- 勤務地
- 本文fingerprint

**結論:**
- **Notionは案件閲覧UI**
- **本体は別DB**
- これが必須

## Q5. 段階的に進めるなら優先順位は？
**優先順位はこれで固定。**

### Phase 1
**原文全件保存**
- まず取りこぼしゼロの基盤を作る
- ここをやらないと後で再処理できない

### Phase 2
**案件候補抽出をRecall重視に変更**
- `project_like / non_project` の二値化
- 現行skip判定を厳しくしない
- 事務、運用、ヘルプデスクも拾う

### Phase 3
**構造化スキーマ拡張**
- 低スキル案件も落ちない項目設計にする
- `job_category` を広げる
- `confidence` を保持する

### Phase 4
**Notion分離**
- 全件Notion登録をやめる
- 高確度・要確認のみ登録
- 残りはDB保持

### Phase 5
**重複統合**
- 月次分析の前に必須
- 同一案件の束ね込みを入れる

### Phase 6
**月次分析バッチ**
- 集計SQLまたはPythonでレポート生成
- Notionにレポートページを作る
