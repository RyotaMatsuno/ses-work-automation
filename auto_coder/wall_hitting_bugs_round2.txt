# 質問1: v6.16 の `名前` 欠落仮説の妥当性

## 結論
**有力仮説です。だが「title欠落なら必ず500」とは断定不可。要確認。**  
通常のAPI設計感覚では **400 Bad Request** が自然です。  
ただし、Notion APIや内部バリデーション経路次第では **500相当の不正系が返る実例はあり得る** ため、今回の観測だけでは否定できません。

---

## 妥当な評価

### 1. 仮説が有力な理由
あなたの発見した差分は大きいです。

- 現行 engineer DB に `名前`(title) がある
- v5.1 は `名前` を参照している
- v6.16 では `名前` 参照が消えていた
- engineer DB の page create で title 未設定はかなり危険

Notionでは DB の title プロパティは実質必須です。  
よって **engineer page create 時に title 未送信** なら、失敗要因として十分強いです。

---

### 2. ただし「500確定」は言い過ぎ
APIとしては普通はこうです。

- 必須項目欠落
- 型不一致
- 存在しない property 名
- 無効な select 値
- 無効な relation page_id

→ まず **400系** が自然

なので、

> 「title 欠落だから 500 が返るはず」

は断定不可です。  
正確には、

> 「title 欠落が内部エラー経路を踏んで 500 を誘発した可能性はある」

までです。

---

## 他に v6.16 で疑うべきパターン

`名前` 以外に、Notion 500/400を起こしうる代表は以下です。

### A. 必須 title 欠落
- 最有力
- 特に engineer DB create で危険

### B. property 名はあるが型が違う
例:
- `名前` に `rich_text` を投げるべきでないのに `title` でない形式
- `select` に `rich_text` 形式を送る
- `relation` に配列でなく文字列を送る

### C. select/status の不正値
- 選択肢が存在しない
- 大文字小文字違い
- 全角半角違い
- v6.16 側で新しい値を作っていた

通常は 400 寄りですが、実装修正によって例外経路が増えていれば要確認。

### D. relation 先 page_id 不正
- 空文字
- UUID形式崩れ
- アクセス権のない page_id
- archive済み / 取得不能 page

これも create/update 失敗要因です。

### E. payload JSON 自体の壊れ
- `None` をそのまま構造に混ぜた
- list/dict の構造不整合
- title array が空
- `properties` のネスト不足

### F. v6.16 で create/update の分岐が壊れた

# 質問2: Queued → 約3分後起動

## 結論
- **Windows Task Schedulerで起こり得る正常寄り挙動**
- ただし **毎回1〜3分遅れるなら要観察**
- `UseUnifiedSchedulingEngine=true` の影響は **十分あり得るが断定不可**
- 1分以内に詰めたいなら **設定見直し余地あり**

---

## どう解釈するか

`schtasks /run` 後に `Queued` になり、数分後に開始するのは異常とは言い切れません。  
Task Scheduler は以下の影響を受けます。

- サービス側の実行キュー
- トリガ再評価
- 既存インスタンス状態
- 電源/アイドル/ネットワーク条件
- Unified Scheduling Engine の内部都合
- タスク履歴やCOM周りの遅延

今回、**最終的に正常実行できている**ので、タスク定義や実行経路が完全破損しているわけではありません。

---

## `UseUnifiedSchedulingEngine=true` の影響
**候補として妥当です。要確認。**

Unified Scheduling Engine は古いタスクより挙動がやや読みにくく、

- 実行要求直後でも即開始しない
- 条件評価後にまとめて開始
- Queued 表示が長め

ということがあります。

ただし、これだけで「19:00が飛んだ」と断定はできません。  
**履歴(Event Log)確認が必要**です。

---

## 1分以内に短縮する設定候補

以下は効果候補です。**恒久変更はCEO確認寄り**。

### 1. `StartWhenAvailable=true` の確認
未実行分をすぐ拾うための設定。  
ただし今回の「Queued短縮」に直結するかは限定的。

### 2. 条件タブの見直し
- `Start only if idle` が有効なら無効化
- `Start only if network available` が有効なら要確認
- `DisallowStartIfOnBatteries` は今回低そうだが一応確認済みでOK

### 3. `MultipleInstancesPolicy`
`IgnoreNew` だと前回残り時に次回が流れます。  
ただし今回のログでは 19:48開始・19:50完了なので、20:00実行には通常干渉しないはず。

### 4. `UseUnifiedSchedulingEngine=false` で再作成/再設定
効果候補としてはあります。  
ただし **タスク再作成レベルの変更** になりやすく、今夜の緊急復旧中に触る優先度は低いです。

### 5. タスクの実行ファイル・開始ディレクトリ・資格情報の健全性確認
起動直前で待たされるケースがあります。  
ただし今回は起動後は正常なので優先度は中。

---

## 業務影響評価

### 1〜3分遅延の影響
**現時点では軽微** と評価してよいです。

理由:
- SESメール案件収集はリアルタイム性より **取りこぼし防止** が重要
- 毎時定期実行で3分以内遅延なら実害は小さい
- バックログ回収可能ならなおさら問題小

### 影響が大きくなる条件
- 1時間制限に近い重処理がある
- 次回実行と重なる
- 受信→通知が厳密な SLA を持つ
- 遅延が「3分」ではなく「たまにスキップ」に化ける

なので評価は、

> **3分遅延自体は許容。スキップ再発の有無が本丸。**

です。

# 質問3: 管理者権限なしで jobz-watchdog 小文字版を実質無効化する手段

## 結論
**タスクそのものを止める/書き換えるのは一般権限では厳しい**です。  
`/change` が denied なら、**タスク定義変更系はほぼ不可** と見てよいです。

「起動先を存在しないパスにする」も、結局タスク変更権限が必要なので **不可** です。

---

## 一般権限でのワークアラウンド案

### 案1: 無視して放置
**現実的にはこれが最も安全**です。

前提:
- CamelCase版 `JobzWatchdog` は正常
- 小文字版は `0x80070002` を吐くだけ
- mail_pipeline本体には直接影響していない

対応:
- 今夜は触らない
- CEO帰還後に管理者で disable/delete

これは十分合理的です。

---

### 案2: 実行先ファイルがユーザー配下なら “空のダミー” を置く
**条件付き。要確認。**

もし小文字版タスクが参照するパスが、
- 自分に書込権限のある場所
- かつ現在ファイル不存在で `0x80070002`

なら、その場所に **何もしないダミー bat/cmd/exe** を置いて成功扱いにできます。

例:
- タスクが `C:\Users\xxx\jobz-watchdog.bat` を叩いていて不存在
- そこに内容ゼロ同然のbatを置く

ただし注意:
- 実行先パスをまず `/query /xml` 等で確認必要
- Program Files や管理者領域なら不可
- 期待する引数/作業ディレクトリ次第で別エラーになる

これは「タスク変更なしで実質無害化」できる唯一の可能性があります。

---

### 案3: イベント/ログ汚染だけ抑える
根本停止ではないですが、現場対応としてはあり。

- 監視対象から除外
- エラー集計で小文字版を ignore
- 引き継ぎメモに「既知ノイズ」と明記

障害切り分け上、**本体障害とノイズを分離**する意味があります。

---

## 非推奨
- タスクファイル直接編集
- `System32\Tasks` 直書き
- レジストリ操作
- ACL変更試行

一般権限ではほぼ無理で、壊すリスクが高いです。
