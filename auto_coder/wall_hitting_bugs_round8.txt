## Cursor指示書
### 目的
`SES_MailPipeline_R7` / Task Scheduler 依存を廃止し、`jobz-command` に mail_pipeline の毎時実行機能を内蔵する。  
要件は **毎時00分実行、二重起動防止、スリープ復帰後catch-up、手動実行API、履歴確認**。

### 実装要件
1. **新規ジョブ定義**
   - job name: `mail_pipeline_hourly`
   - command: 既存の `wd_mail_pipeline.bat` もしくは直接 `python mail_pipeline.py`
   - working directory を明示
   - stdout/stderr をログファイルへ保存

2. **scheduler**
   - 毎時 `HH:00` に実行
   - 秒単位ズレ許容は ±60秒程度でよい
   - 前回予定時刻を state に保存すること

3. **catch-up**
   - プロセス起動時
   - スリープ復帰後の最初のtick時
   に、前回予定時刻以降で未実行の hourly slot があれば **1回だけ補填実行**
   - 上限は直近1〜3スロット程度でよい
   - 無限連続catch-upは禁止

4. **二重起動防止**
   - mail_pipeline 実行中なら次回 scheduled run は skip
   - lock file または in-memory + pid/state 管理
   - 手動APIからの実行も同じ lock を使う

5. **API**
   - `POST /jobs/mail_pipeline/run`
   - `GET /jobs/mail_pipeline/status`
   - `GET /jobs/mail_pipeline/history`
   - status に以下を含める
     - running
     - last_started_at
     - last_finished_at
     - last_exit_code
     - next_due_at
     - last_scheduled_slot

6. **永続state**
   - `job_state/mail_pipeline_hourly.json`
   - 項目:
     - last_scheduled_slot
     - last_started_at
     - last_finished_at
     - last_exit_code
     - last_success_at

7. **ログ**
   - `logs/mail_pipeline_hourly/YYYY-MM-DD.log`
   - 1実行ごとに開始・終了・exit code・scheduled slot を記録

8. **異常系**
   - exit code != 0 なら失敗記録
   - 次の毎時実行は通常通り継続
   - 復帰直後ネット不安定を考慮し、**起動側ではなく mail_pipeline 側で軽い retry を検討可能**だが今回は必須ではない

9. **Task Scheduler廃止**
   - mail_pipeline用のTask Schedulerは disable 推奨
   - ただし切り戻し用に削除ではなく disable

10. **受け入れ条件**
   - 手動APIで正常実行できる
   - 擬似的に state を過去時刻へ戻して catch-up が1回動く
   - 実行中に再度runしても二重起動しない
   - status/history で運用確認できる
