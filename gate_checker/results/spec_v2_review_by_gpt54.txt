## 1. SPECとTASKS、CLAUDE.mdとの整合性

### 良い点
- **モデル不在時 fallback gpt-4o** は3文書で一致。
- **exit code 2 = CostGuard停止 / 日次上限超過** も一致。
- **装置2閾値**（軽 `$0.025` / 中 `$0.10` / 重 `$0.15`）は一致。
- **装置3の重複起票防止キー `(yyyymmdd, block_type, phase)`** は一致。
- **`agreement_checker` は触らない**方針も一致。
- **MODEL_PRICING 検証を Phase 9 で実施**という運用も一致。

### 気になる不整合
1. **CostGuard判定順序の記載が揺れる**
   - SPEC §4-2 は DAILY_CALL_LIMIT を前面に出している。
   - SPEC §7 は **「API呼び出し前: `can_spend()` → DAILY_CALL_LIMIT」** と読める。
   - TASKS Phase 2 は **`check_daily_limit()` を先に書いて、その後 CostGuard拒否時** としている。
   - ここは実装順によって **block_type が `daily_limit` になるか `costguard` になるか** が変わり、通知内容や Notion task_id に影響します。

2. **装置3の重複時挙動**
   - SPEC §6-2 では「**既存ページにcounter追記**」とある。
   - TASKS Phase 6 では「**起票・通知ともスキップ**」とある。
   - 既存ページ更新までやるのか、完全スキップなのか、実装の解釈差が出ます。

3. **結果JSONの `original_model`**
   - SPEC §3-3 で「fallback_used / original_model を記録」とあるが、fallbackしない場合に `original_model` を元モデルで埋めるのか `None` にするのかが実例JSONでは元モデルが入っている。
   - TASKSはこの詳細に触れていないので、テスト期待値が揺れる可能性があります。

4. **Week1中のレビュー経路**
   - TASKS Phase 0 / 11 では `gate_check.py` のハルシネーション問題回避のため、**`spec_v2_review_by_gpt54.py`** 等の別経路を使う。
   - しかしSPEC本体はその運用例外をほぼ持っていない。開発運用文書としては問題ないですが、「このSPEC自体をこのツールでレビューする」オペレーションはまだ不安定です。

**結論**: 大枠整合しているが、**判定順序・重複時挙動**は明文化を揃えるべきです。

## 2. 仕様の論理的破綻

致命的破綻はありません。  
ただし、以下は軽い論理矛盾または未定義です。

### A. 判定順序未統一
前述の通り、**`can_spend()` と `check_daily_limit()` の順番**が文書間で揺れています。  
このままだと同じ状況でも、
- 実装A: `daily_limit`
- 実装B: `costguard`
になりえます。

### B. `available_models` 取得失敗時の扱い
TASKSでは
- `OpenAI.models.list()` 失敗時は **空 set + WARNINGログ**
とあります。  
一方 `resolve_model()` は **空 set** を渡されると全モデルが fallback になります。これは仕様としては筋が通っています。  
ただしSPEC §3-2 の docstring では `available_models is None` の場合通常解決です。実装者が `None` を渡すと、未知モデルでも fallback されません。  
**None と 空set の意味差**は要注意です。

### C. `estimate_cause()` の優先順位
```python
if daily_usd > DAILY_HARD_USD * 0.9 and daily_calls < 5:
if daily_calls >= DAILY_CALL_LIMIT:
if monthly_usd > MONTHLY_USD * 0.9:
```
この順序だと、複数条件を同時に満たすケースで「単発コスト過大」が優先されます。  
たとえば月末で月次逼迫かつ当日件数超過でも、先頭条件次第で説明がずれる可能性があります。致命的ではないですが、原因分析としてはやや粗いです。

### D. 集約ロジックの実現性
「**同日・同block_type・同phaseの2回目以降は集約のみ（既存ページにcounter追記）**」は、**task_id が日単位固定**なので実装可能です。  
ただし Notion API 側で「既存ページ検索→更新→counter追記」が必要になります。TASKSの記述はそこまで具体化されていないため、Week1で本当にやるならもう少し処理手順を詰めたいです。

## 4. 通知集中による事故リスク（LINE月200通枠）

ここは**v2.1でかなり改善**されています。

### 良い点
- SPEC §13で**優先順位と残通数閾値**が明文化された。
- **装置3**を最優先にしたのは妥当。
- **装置2**を「同日同phase同class 1回」にしたのもよい。
- **fallback通知**も「モデル名キーで1日1回」で抑制している。

### 残るリスク
1. **モデル不在時の通知スパム**
   - `OpenAI.models.list()` 失敗→ available_models 空set → 全phase fallback → 各phaseで通知、となる可能性があります。
   - 仕様上は「1日1回（モデル名キー）」ですが、**phaseをまたいで同一fallbackモデル通知がどこまで集約されるか**は実装依存です。

   - 安全装置としては正しいですが、残10通から複数phaseでブロックすると、月末に枠を使い切る可能性はあります。
   - とはいえ「CostGuard停止」は重要度が高く、ここを削るより装置2/4を削る設計のほうが合理的です。

3. **NG通知と松野確認通知の競合**
   - `needs_human_review()` が強めなので、`requirements / design / test` は常に確認推奨。
   - そこにモデルfallbackや装置2が重なると、同一targetに複数通知イベントが発生し得ます。
   - §13の優先順位で抑制する前提ですが、**同一実行で1通にまとめる設計**まではSPECにありません。

**評価**: 重大事故リスクは下がったが、**“同一実行の通知統合”がないため、まだやや多い**です。Week1なら許容範囲。

## 5. 装置2・装置3の発動条件の適切さ

  - 軽 `$0.025`
  - 中 `$0.10`
  - 重 `$0.15`

### 装置3
発動条件:
1. `common.ledger.can_spend()` が False

**総合**

- Week1としては「鈍感すぎる」より安全側でよいです。

## 6. フェーズ別モデル名の妥当性

### 結論
- **「モデル名が実在するか不安」へのSPEC上の対応はある**ので、この観点は概ねOK。
- ただし、**運用初回前に Phase 9 の pricing確認を前倒ししたほうがよい**です。

## 8. agreement_checkerを触らない選択の妥当性

**Week1スコープとしては妥当**です。

### 理由
- 既存 `run_dual_review()` は **GPT-4o + Gemini** の2系統で、ここに同時変更を入れると影響面が広すぎる。
- v2.1の中心は gate_checker 側の

  - CostGuard
  - 装置2/3
 なので、対象を絞るのは正しいです。
- SPEC §7 で **Geminiは ledger.record済み** と最低限の被覆を確認しているのもよい。

### ただし
- `call_gpt4o()` の中身だけフェーズ別モデルに変えて、関数名は据え置きというのは、**可読性・保守性では負債**です。
- これはSPEC自身も「Week2で命名整理予定」と認識しているため、今は許容可能。

**結論**: 触らない判断は妥当。ただし**“不変”ではなく“当面凍結”**として扱うべきです。
