## Q2. 「API used = 0%」の解釈は正しいか?

   - これが最有力です。

2. **特定機能・特定モデル時だけAPIキーが使われる**  
   - 普段はAuto枠、ある条件でだけBYOK、という仕様の可能性。

3. **UIの%表示が別母数**  
   - Total 6% / Auto 8% の不一致がある以上、**表示定義の誤読リスク**がある。

4. **外部ベンダー側課金がCursor UIに即時/完全反映されない可能性**  
   - 断定は避けるべき。  
   - **AnthropicコンソールのUsage/Billing**で実測確認が必要。

**正しい言い換え**  
- ×「Anthropic Key ONは機能していない」  
- ○「現時点では、Cursor UI上はAnthropic Key経由使用が観測されていない」

## Q3. 線形換算(6%/7日→26%/月)の前提は妥当か?

**結論: 弱い。参考値にはなるが、確定判断の根拠としては不足。**

### 問題点
1. **サンプルが短い**  
   - 7日だけでは、月内の繁閑を拾えません。

2. **月末集中・納期前スパイクを無視**  
   - 実装は均等消費ではなく、**仕様確定後に一気に使う**ことが多い。

3. **A/C/Dが進むほどCursor使用量が増える可能性**  
   - 3点セット作成後、実装フェーズに入ると消費が跳ねるのが普通。

4. **そもそも元データが不整合**  
   - 6%と8%の意味が揃っていないなら、線形換算以前の問題。

### どう扱うべきか
- 線形換算は **ベースケース** に留める。  
- 追加で **繁忙ケース** を置くべきです。例:
  - 平常: 26%
  - 月後半2倍: 40〜50%
  - 納期前3倍: 60〜80%
- それでもPro内なら安心度が上がる。

**必要な管理**
- しきい値管理:  
  - **月中50%超** → 要観察  
  - **月中70%超** → Key/Ondemand方針再確認  
  - **月中85%超** → フォールバック発動判断

## Q4. Anthropic Key OFFのデメリットは?

**あります。主に“非常時の継続性”が落ちます。**

### デメリット
1. **フォールバックを失う**  
   - Pro枠不足やCursor側障害時に、**BYOKで継続する逃げ道**が減る。

2. **特定モデル/機能が使えない可能性**  
   - BYOK前提のモデルや挙動があるなら制約になる。

3. **復旧時の手間**  
   - いざ必要になった時に再設定・検証が要る。

### ただし
- 今の優先が **「事故課金防止」** なら、OFFは合理的。  
- その代わり、**再ON手順をメモ化**しておくべきです。

**推奨**
- **常時OFF**でもよい。  
- ただし `障害/枠不足時の再ON手順` を1枚にして残す。  
- 可能なら **Anthropic側にも予算アラート** を設定。

## Q5. 論点B確定後にA/C/Dへの影響が「なし」は本当に正しいか?

**結論: 課金経路は独立だが、運用・品質・負荷は独立ではない。  
「完全に影響なし」は言い過ぎ。**

### つながり
1. **A/C/DがCursor消費量を増減させる**
   - gate_checkerや二次壁打ちの設計次第で、  
     - 事前整理が効いてCursor往復が減る  
     - 逆にタスク量が増えてCursor消費が増える  
   - よって **論点Bの使用量予測** に影響します。

2. **モデル差でプロンプト設計が変わる**
   - OpenAI向けに作ったSPEC/TASKSが、Sonnet実装で最適とは限らない。  
   - つまり **ドキュメント設計の最適化対象** が変わる。

3. **全体AI予算は連動**
   - Cursorは安くても、A/C/DのOpenAI APIが増えれば全体コストは増える。  
   - 経営判断は総額で見るべき。

4. **品質責任の分散**
   - 企画はOpenAI、実装はSonnet、で責任境界が曖昧になりやすい。  
   - 不具合時の原因切り分けコストが増える。

**正しい表現**
- ×「影響なし」  
- ○「請求経路は独立。ただし、使用量・プロンプト設計・全体コストには間接影響あり」
