# GPT-5.4が見落とした論点(3つ以上)

- 解放条件が「警告0件」だと、運用が“安全化”ではなく“警告を出さない化”に歪む: GPT-5.4は上限値・独立性・機械判定性を中心に見ており、KPIが人間行動を歪める問題まで踏み込んでいない / 事業上は、警告を減らすためにタグ付けを避ける、差分を小分けにする、risk_scoreを低く出す、危険変更を通常変更に偽装する、といった「安全装置回避インセンティブ」が発生する。

- 「risk_score>=0.7」の算出器そのものが未設計: GPT-5.4は“ハイリスク判定が機械判定不能”とは指摘したが、最終案で導入されたrisk_scoreの算出ロジック・責任主体・検証方法までは批判していない / risk_score算出モジュールが新たな単一障害点になり、誤検知・見逃し・恣意的調整が起きる。安全設計のはずが、未検証のスコアリングAIに安全判断を委譲する構造になる。

- 二次壁打ちにminiを使うと「チェッカーの品質が被チェック対象以下」になる: GPT-5.4は一次/二次の型番系列を分けるべきとは指摘したが、二次を軽量モデルに落とすことの構造的劣化は十分に見ていない / final_gateや法務・契約・送信系の高リスク領域で、より弱いモデルがより強いモデルの出力を監査する逆転が起きる。これはコスト削減にはなるが、安全保証としては弱い。

- 「モデル分離」と「能力分離」は別問題: GPT-5.4は系列分離を独立性確保として評価したが、gpt-5.4とgpt-5.4-miniは同一ファミリー由来のバイアス・同種の失敗モードを共有する可能性がある / 形式上は二重化していても、実際には同じ盲点で同時に誤る可能性がある。独立性を担保するなら、モデル系列だけでなくプロンプト方針、評価観点、禁止基準、出力形式も分離すべき。

- 二層化により「金額上限でAI全体が止まるが、業務上必要なgate_checkerも止まる」運用不整合が残る: GPT-5.4はDAILY_CALL_LIMITとCostGuardの重複を批判したが、二層化後に起きる“停止順序の業務影響”までは見ていない / 月末や障害対応時に他AI処理がCostGuardを消費し、肝心のgate_checkerが止まる。すると安全確認が必要な場面ほど安全装置が使えなくなる。

- gate_checker専用予算と一般AI予算の分離がない: ジョブズ案は層1を回数、層2を金額とするが、gate_checkerに最低保証枠がない / 営業文面生成・調査・要約などでAI予算を食い潰すと、安全系のgate_checkerが巻き添え停止する。安全装置は「余ったら使う」ではなく「最後まで残す」べき。

- worst-case単価$0.070/callが本当の最悪値とは限らない: GPT-5.4は平均単価ではなくworst-caseで見ろと指摘したが、そのworst-caseの妥当性検証までは踏み込んでいない / 月末集中、障害対応、契約締結直前、法務確認、freee連携、送信前チェックが同日に重なると、長文コンテキスト・再試行・二次壁打ち・ログ要約で単価が跳ねる。$0.070を固定最悪値にすると、実際の極値に耐えない可能性がある。

- 「7日連続」の解放条件は、営業会社の業務周期と噛み合わない: GPT-5.4は時限解放より実績連動解放を提案したが、7日という観測窓の妥当性は見ていない / SES営業は平日偏重、月末偏重、締め日偏重、案件波動がある。閑散週に7日クリアして上限解放され、繁忙週に急に破綻するリスクがある。

- call数上限は安全リスクの代理指標として粗すぎる: GPT-5.4はDAILY_CALL_LIMITの金額重複を見たが、「回数」と「危険度」のズレは十分に批判していない / 低リスクの短文チェック100回より、高リスクの契約・送信・請求関連1回の方が危険な場合がある。回数上限では真のリスク制御にならない。

- タグ含有判定は抜け道が多い: 最終案では「売上/契約/法務/freee/送信系タグ含有」が二次壁打ち条件だが、タグ付け漏れ・表記揺れ・暗黙の業務変更を拾えない / “請求書”はfreeeタグに入るのか、“候補者情報”は個人情報タグに入るのか、“Slack通知”は送信系か、など曖昧領域が多い。結果として二次壁打ち対象の取りこぼしが残る。

- 差分>300行は回避可能で、かつ危険度と相関が弱い: GPT-5.4はimplementation NG時のみの逆設計を批判したが、最終案の「差分>300行」条件自体の弱さは未指摘 / 20行の認可条件変更、1行の送信先変更、環境変数名の変更、SQL条件変更の方が危険な場合がある。一方で大量のコメント修正は300行を超えても低リスク。

- 「警告0件」は成熟度の証拠ではなく、検出器が鈍い証拠かもしれない: 解放条件が警告数だけを見ると、検出精度の悪化と安全性向上を区別できない / プロンプト変更やタグ判定変更で警告が出にくくなっても、解放条件を満たしてしまう。安全装置の有効性を測るには、警告数だけでなくサンプリング監査・偽陰性率・レビュー後修正率を見る必要がある。

# 長期視点での問題(法人化・スケール・人員増加観点)

- 人員増加時に「誰が上限を消費したか」が見えない: 個人別・チーム別・業務別の予算分解がないと、営業担当Aの大量利用でCEOの重要なgate_checkerが止まる / 組織化後は利用者単位のquota、優先度、監査ログが必要になる。

- 法人化後は「安全装置の停止」が内部統制上のリスクになる: 現状は2名会社なので手動補完できるが、法人化後に請求・契約・個人情報・外部送信が増えると、gate_checker停止時の代替手順が必要 / 止まった場合に「誰が承認するか」「何を禁止するか」「後追い監査するか」が未設計。

- CostGuard $140/月のままスケールすると、安全系と生産系AIが同じ財布を奪い合う: 人員が増えるほど通常AI利用が増え、安全系gate_checkerの予算を圧迫する / 将来的には「安全系AI予算」「営業生産AI予算」「実験予算」を分けるべき。

- mini二次壁打ちは、事業の重要度が上がるほどリスクが顕在化する: 2名会社の初期運用では許容できても、法人化後に契約金額・個人情報量・取引先数が増えると、低品質チェッカーによる見逃しコストが跳ね上がる / 重要度に応じて二次モデルを昇格させる設計が必要。

- 7日連続基準は組織が増えるとゲーミングされやすい: メンバーが増えると「今週は解放したいから危ない変更は来週に回す」「警告が出そうな作業はタグを外す」などの行動が起きる / 解放条件は警告0件ではなく、一定件数以上の正常運用実績、ランダム監査合格率、重大変更のレビュー完了率を見るべき。

- risk_score算出モジュールの保守者問題が発生する: 法人化後、業務が増えるとリスク分類も増える。契約、請求、候補者情報、顧客情報、稟議、外部送信、アカウント権限などを誰が更新するかが問題になる / 放置すると古いrisk_scoreが現場実態とズレ、危険変更を低リスク扱いする。

- 業務ピークに弱い: SES営業は月末、契約更新、要員提案、請求、稼働開始・終了のタイミングで処理が集中しやすい / 日次上限だけだとピーク日に安全確認が詰まり、業務停止または手動バイパスが発生する。

- 監査証跡が足りない: gate_checkerが何を止め、何を通し、誰がoverrideしたかが残らないと、後で事故調査できない / 法人化後は「判断ログ」「入力要約」「モデル名」「risk_score根拠」「二次壁打ち結果」「人間承認者」を残す必要がある。

- バイパス手順が未定義: 上限到達時・CostGuard到達時・API障害時・モデル障害時にどう業務継続するかがない / 人員増加後は各自が勝手に手動送信・手動契約確認を行い、統制が崩れる。

- 上限値が会社規模に対して固定的すぎる: 30/60/90/110 call/dayは2名体制では妥当でも、5名、10名、20名ではすぐ機能しなくなる / 「会社全体の固定上限」ではなく「人数・案件数・高リスク操作数に応じた動的quota」が必要。
