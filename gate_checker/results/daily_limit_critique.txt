in=1065 out=2548 cost=$0.0409

# 提案の脆い前提(3つ以上)
- 前提: 平均単価を **$0.0313/call** で固定して `DAILY_CALL_LIMIT` の1日上限コストを計算している / 崩れた場合の損失: 実運用で **gpt-5.4 の比率** や **final_gate($0.070/call)** が集中した日に、段階4=**240回**でも **$7.51** を超えて **$8/日 CostGuard** に接近・到達する。平均単価前提が崩れると「94%で安全」という説明が虚偽になる。
- 前提: `gate_checker` 系だけを `DAILY_CALL_LIMIT` の対象にしても、実質的なスパイクはそこだけで起きる / 崩れた場合の損失: 障害日や大規模リファクタ月では **gate_checker + matching_v3 + mail_pipeline** が同時増加し、`DAILY_CALL_LIMIT` は守っても **実課金総額** は想定超過する。局所防御で全体予算逸脱を見逃す。
- 前提: 段階1=**50回/日** は「異常検知時のスパイク対応」と「初期運用の作業継続」を両立できる / 崩れた場合の損失: 初期Week1で要件変更・設計見直し・テスト再実行が重なると、**50回**は低すぎて通常業務を止める。一方で、もし高単価フェーズが少ない構成なら **50回でも十分すぎる** ため、異常検知用途としては粗すぎる。
- 前提: 二次壁打ちの発動条件「ハイリスク判定」「NG判定」が機械判定可能 / 崩れた場合の損失: 人手判定に依存すると、発動漏れで **共謀リスク対策が形骸化** するか、逆に広く拾いすぎて **月発動3回→10回超** に膨らむ。追加コスト **$0.66/月** は容易に崩れる。
- 前提: 月回数 **113回/月** の確定前提が、日次上限設計にもそのまま有効 / 崩れた場合の損失: 月間合計が同じでも、実際は月末集中・障害集中で **1日偏在** する。月平均 **113/30=3.77回/日** という平常値に対し、上限 **50/100/180/240** はギャップが大きく、異常検知の感度が低い。
- 前提: CostGuard **$8/日, $140/月** と `DAILY_CALL_LIMIT` の二重防御は独立した意味を持つ / 崩れた場合の損失: 段階4=**240回= $7.51/日** は **CostGuardの94%** であり、実質的に同じ閾値の言い換え。片方が誤設定でもう片方が救う構造になっていない。

# 数値設計の問題点
- 段階値の妥当性  
  - 段階1=**50回/日** は、月前提 **113回/月** に対して **13.3日分の平均消費量** を1日で許容する設定。異常検知としては鈍い。  
  - 段階2=**100回/日** は月間想定の **88.5%** を1日で使える水準。Week2の安全装置1追加だけでここまで上げる根拠が弱い。  
  - 段階3=**180回/日** は月間想定の **159%/日**。もはや「日次制御」より「上限撤廃に近い」。  
  - 段階4=**240回/日** は **CostGuard $8/日** に対して **$7.51/日**。防御線としてほぼ重複で、事故の早期検知には使えない。  
  - しかも計算は全て **平均単価 $0.0313/call** 依存。高単価日シナリオの worst-case を見ていない。例として **final_gate($0.070)** だけで **50回** なら **$3.50/日**、**100回** なら **$7.00/日**。段階2でも CostGuard にかなり近い。
- 二次壁打ち発動条件の妥当性  
  - `design`: **全件発動(8回)** は機械判定不要で明快。ただし、なぜ `requirements` や `test` は対象外なのかの線引きがない。共謀リスクを本当に懸念するなら、設計だけ全件は中途半端。  
  - `final_gate`: **ハイリスク判定時のみ(月3回想定)** は曖昧。何をもってハイリスクとするか、**スコア閾値・禁止語・差分規模・金額影響** の定義がない。機械化できない条件は運用崩壊する。  
  - `implementation`: **NG判定時のみ(月2回想定)** は逆。共謀リスク対策なら「一次判定が通したが危険」ケースを拾うべきで、**NGのみ再審** は「危険な通過」を取りこぼす。品質補助としても設計不整合。  
  - 追加コスト **$0.66/月** は発動回数が固定前提。実際に `implementation` の NG が **2回→10回** になる月は普通にありえる。
- コスト試算の漏れ  
  - **高単価偏在日** の worst-case 試算がない。平均単価だけでは日次上限の安全性は証明できない。  
  - `DAILY_CALL_LIMIT` 対象外の **matching_v3** の日次コスト上振れが未評価。  
  - 「二次壁打ち」に使う **別モデルの型番・単価** が明示されていない。共謀リスク対策なのに、同系統・同価格帯モデルなら独立性が弱い。  
  - **リトライ、タイムアウト、再送、空振り呼び出し** の課金が含まれていない。小規模体制(2名)ほど運用事故時の再送率が上がる。  
  - 月上限 **$140** と現行見積 **$4.60/月** の乖離が大きすぎる。予算余剰が大きいのに、なぜ日次だけ神経質で月次ガバナンスが粗いのか、設計思想が不整合。

# 別観点での代替案
| 項目 | 提案案 | 代替案 | 根拠 |
|---|---|---|---|
| 段階値 | 50 → 100 → 180 → 240 | **20 → 40 → 80 → 120** | 月想定 **113回/月** に対し、提案値は過大。異常検知用途なら平常の **5〜10倍** 程度で十分。120回/日でも平均 **31.8日分/日** ではなく、約 **31.8倍平常日量** ではない点に注意すべきだが、少なくとも 240 より早く異常を止められる。コストも平均単価なら **$0.63 / $1.25 / $2.50 / $3.76**。 |
| 対象範囲 | `gate_checker` 系のみカウント | **二層化**: ①`gate_checker` 個別上限、②AI全体共通の金額上限 | 単価差があるので回数閾値は個別でよいが、**総課金** は別問題。`mail_pipeline($0.0003)` は回数でなく金額監視、`matching_v3` は別枠でも **全体$上限** に包含すべき。 |
| 二次壁打ち条件 | design全件 / final_gateはハイリスク時 / implementationはNG時 | **ルール化**: design全件、final_gateは`risk_score>=0.7`または`売上/契約/法務`タグ含有時、implementationは`差分>300行`または`権限/課金/個人情報`変更時 | 機械判定可能な条件に落とさないと運用不能。特に implementation の「NG時のみ」は危険な通過を拾えないので、**変更特性ベース** に変えるべき。 |
| 二次壁打ち条件 | 共謀リスク対策だが別モデル定義なし | **一次と二次で型番分離を明示** | 「複数モデル壁打ち」が目的なら、同一系統・同一設定では独立性が弱い。最低でも **一次=gpt-5.4、二次=gpt-5.4-mini または別系列** などの分離方針が必要。 |
| 段階解放条件 | Week1/2/3/1ヶ月で時限解放 | **実績連動解放**: 7日連続で`上限消費率<30%`かつ`誤検知0件`で次段階 | 安全装置の追加だけでは、実運用安定を証明できない。**日付ではなく観測実績** で解放すべき。 |
| CostGuardとの関係 | 日次回数上限 + CostGuard $8/日 | **役割分離**: `DAILY_CALL_LIMIT` は異常検知、CostGuard は最終停止。段階4でも CostGuard の **60%以下** に抑える | 現状の段階4=94%は役割重複。異常検知はもっと手前で止めるべき。例えば平均単価基準でも **150回=$4.70** 程度までに抑える方が二重防御として意味がある。 |

# 確信度
**4/5**。数字上の矛盾と運用条件の曖昧さは明確だが、`matching_v3` の単価・実トラフィック不明のため全体最適の断定は1段留保。
